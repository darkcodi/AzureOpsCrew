# Техническое задание

## Рефакторинг фаза 2

### Переход от мультиагентного чата к полноценному execution engine для сложных задач

---

# 1. Цель фазы

Реализовать второй этап эволюции мультиагентной системы: превратить текущую архитектуру из **role-based orchestration + structured workflow** в **полноценную систему выполнения сложных задач**, способную:

* принимать одну большую задачу от пользователя;
* самостоятельно строить план выполнения;
* декомпозировать задачу на подзадачи;
* назначать подзадачи подходящим агентам;
* выполнять подзадачи пошагово;
* по результатам шагов перепланировать дальнейшие действия;
* сохранять промежуточные результаты и артефакты;
* останавливаться на контрольных точках;
* запрашивать approve пользователя перед критическими действиями;
* завершать задачу только после проверки результата.

Главная цель фазы 2 — добиться поведения, близкого к mature agent systems:
не просто “чат с агентами”, а **система, которая умеет доводить сложную задачу до результата через планирование, исполнение, наблюдение и перепланирование**.

---

# 2. Контекст

Фаза 1 уже должна была внедрить:

* run protocol;
* роли Manager / DevOps / Developer;
* evidence-first workflow;
* approval policy;
* базовую orchestration logic;
* service registry;
* retrieval для статического контекста;
* structured logging;
* incident workflow;
* code fix / PR workflow.

Фаза 2 должна использовать эти результаты как фундамент и не переписывать всё с нуля без причины.

---

# 3. Проблема, которую решает фаза 2

Даже после фазы 1 система может оставаться ограниченной:

* умеет работать только по заранее описанным шаблонам;
* плохо ведёт длинные задачи;
* не умеет качественно декомпозировать крупную задачу;
* не умеет сохранять и использовать промежуточный прогресс;
* не умеет адаптировать план на основе новых данных;
* не умеет работать как долгоживущий исполнитель;
* может повторяться, зацикливаться или терять нить расследования;
* не умеет явно оперировать подзадачами, зависимостями и артефактами.

Фаза 2 должна устранить эти ограничения.

---

# 4. Главный результат фазы 2

После завершения работ система должна уметь брать задачу вида:

* “Разберитесь, почему сайт не работает, и исправьте”
* “Проверьте прод, найдите причину деградации и предложите безопасное решение”
* “Поймите, почему падает deploy, исправьте проблему в коде и подготовьте выкладку”
* “Разберитесь в проблеме по инфраструктуре end-to-end”

и выполнять её в таком режиме:

1. построить initial plan;
2. разбить задачу на подзадачи;
3. определить последовательность и зависимости;
4. выполнять подзадачи через подходящих агентов;
5. сохранять результаты каждого шага;
6. по мере появления новых данных корректировать план;
7. эскалировать пользователю только в нужных местах;
8. запрашивать approve перед рискованными действиями;
9. доводить задачу до результата с явной верификацией.

# 5. Обязательные архитектурные улучшения

## 5.1. Внедрить Task Execution Engine

Нужно добавить в систему полноценный слой исполнения задач.

Этот слой должен быть ответственен за:

* создание root task;
* декомпозицию задачи;
* хранение task tree;
* выбор следующей подзадачи к выполнению;
* отслеживание статуса задач;
* обработку зависимостей;
* перепланирование;
* сохранение артефактов;
* остановку на approval checkpoints;
* завершение root task.

Execution engine не должен зависеть от “болтовни в чате” как основного механизма координации.
Чат — это интерфейс общения с пользователем.
Основной контур выполнения должен жить в состоянии задач, а не только в текстовых сообщениях.

---

## 5.2. Внедрить иерархическое дерево задач

Нужно реализовать структуру задач с поддержкой:

* root task;
* child tasks;
* nested subtasks;
* parent-child связи;
* dependencies between tasks;
* task ordering;
* branch-specific tasks.

Минимальная модель задачи должна включать:

* `task_id`
* `run_id`
* `parent_task_id`
* `title`
* `description`
* `task_type`
* `assigned_agent`
* `status`
* `priority`
* `depends_on`
* `goal`
* `inputs`
* `artifacts`
* `evidence_refs`
* `result_summary`
* `created_at`
* `updated_at`
* `started_at`
* `completed_at`
* `retry_count`
* `replan_count`

Допустимые статусы задач определить явно, например:

* `created`
* `ready`
* `blocked`
* `planned`
* `running`
* `waiting_for_tool`
* `waiting_for_dependency`
* `waiting_for_approval`
* `waiting_for_user_input`
* `succeeded`
* `failed`
* `cancelled`
* `skipped`

---

## 5.3. Реализовать planner / executor split

Текущие агенты не должны одновременно выполнять все функции без структуры.

Нужно ввести логическое разделение:

### Planner layer

Отвечает за:

* понимание root goal;
* построение initial plan;
* декомпозицию на подзадачи;
* обновление плана после новых данных;
* выбор стратегии выполнения.

### Executor layer

Отвечает за:

* выполнение конкретной подзадачи;
* tool calls;
* сбор evidence;
* подготовку конкретных outputs;
* возврат результата execution engine.

Planner и Executor могут быть реализованы либо:

* как разные логические режимы одного Manager-агента,
* либо как отдельные агенты/системные этапы,
* либо как отдельные components orchestration layer.

Но разделение обязанностей должно быть явным в архитектуре.

---

## 5.4. Реализовать динамическое перепланирование

Система должна уметь обновлять план при изменении картины мира.

Примеры:

* изначально казалось, что проблема в инфраструктуре, но логи показали bug в коде;
* изначально был один suspect service, но выяснилось, что проблема в downstream dependency;
* initial remediation plan не сработал, нужен альтернативный план.

Нужно реализовать:

* `initial_plan`
* `current_plan`
* `plan_revision`
* `replan_reason`
* `replanned_by`
* `replan_diff`

Система должна уметь:

* пересобирать следующие шаги;
* закрывать устаревшие подзадачи;
* создавать новые подзадачи;
* сохранять историю изменений плана.

---

# 6. Execution Loop

Нужно реализовать явный цикл выполнения, а не просто последовательность чатовых ответов.

Минимальный execution loop должен работать так:

1. выбрать текущую готовую задачу;
2. понять цель подзадачи;
3. проверить, есть ли все входные данные;
4. решить, нужен ли tool call / retrieval / handoff;
5. выполнить шаг;
6. сохранить результат;
7. обновить статус подзадачи;
8. определить, требуется ли:

   * следующая подзадача,
   * перепланирование,
   * уточнение у пользователя,
   * approval,
   * завершение;
9. перейти к следующему шагу.

Этот loop должен быть реализован как системный runtime mechanism, а не “надеяться, что LLM сама так будет думать”.

---

# 7. Checkpoints и контроль над автономностью

Нужно реализовать систему контрольных точек.

## 7.1. Типы checkpoints

Поддержать как минимум:

* `approval_checkpoint`
* `risk_checkpoint`
* `human_input_checkpoint`
* `replan_checkpoint`
* `verification_checkpoint`

## 7.2. Обязательные checkpoint-сценарии

Система обязана останавливаться минимум в следующих случаях:

* перед deploy;
* перед rollback;
* перед изменением production configuration;
* перед перезапуском production workload;
* при высокой неопределённости;
* если есть несколько конкурирующих гипотез;
* если evidence недостаточно для безопасного действия;
* если remediation plan меняет данные/систему;
* после создания PR, если следующий шаг — merge/deploy.

---

# 8. Budget rules и anti-loop механизмы

Нужно защитить систему от бесконечной работы и бессмысленного хождения по кругу.

Реализовать ограничения:

* максимальное число шагов на run;
* максимальное число шагов на одну подзадачу;
* максимальное число перепланирований;
* максимальное число подряд “безрезультатных” tool invocations;
* максимальное число подряд non-progress turns;
* timeout на этап;
* общий timeout на run;
* правила эскалации при исчерпании бюджета.

Нужно явно определить, что такое “progress”.
Например:

* получен новый evidence;
* выполнен tool call с полезным результатом;
* narrowed hypothesis space;
* создан PR;
* выполнена verification;
* получен approval;
* изменён статус задачи;
* завершена подзадача.

Если прогресса нет, система не должна продолжать бесконечно.

---

# 9. Memory layer и execution journal

Нужно внедрить память выполнения задачи.

Это не “человеческая память” и не просто chat history.

Нужен **execution journal**, который хранит:

* какие шаги уже сделаны;
* какие гипотезы проверены;
* какие гипотезы отброшены;
* какие tool results уже получены;
* какие артефакты созданы;
* какие изменения уже были предложены;
* какие approvals уже были запрошены и с каким результатом;
* что уже знает каждый агент в рамках run.

Нужно реализовать минимум 3 слоя памяти:

## 9.1. Run memory

Память всей root задачи.

## 9.2. Task memory

Память отдельной подзадачи.

## 9.3. Shared evidence memory

Хранилище доказательств и результатов, на которые могут ссылаться все агенты.

---

# 10. Artifact store

Нужно ввести явное понятие артефактов.

Артефакты могут быть:

* логи;
* summaries tool outputs;
* KQL query results;
* health snapshots;
* deployment diagnostics;
* code diff;
* branch info;
* PR link;
* verification report;
* incident summary;
* rollback plan;
* risk assessment.

Каждый артефакт должен иметь:

* `artifact_id`
* `run_id`
* `task_id`
* `artifact_type`
* `source`
* `created_by`
* `created_at`
* `content_ref`
* `summary`
* `tags`

Агенты не должны пересказывать всё текстом каждый раз.
Они должны уметь ссылаться на артефакты и строить новые шаги на их основе.

---

# 11. Reflection / self-check layer

Нужно внедрить ограниченный слой рефлексии качества шага.

После важных execution steps система должна уметь делать короткую self-check процедуру:

* достаточно ли evidence;
* нет ли противоречий;
* не требуется ли replan;
* действительно ли текущая гипотеза сильнее альтернатив;
* выполнена ли цель подзадачи;
* нужен ли handoff другому агенту.

Важно: это должна быть **короткая, структурированная проверка**, а не бесконечное “размышление ради размышления”.

Нужен строгий формат, например:

* `step_outcome`
* `confidence`
* `open_questions`
* `need_replan`
* `next_recommended_action`

---

# 12. Handoff framework

Нужно внедрить формализованный handoff между агентами.

Сейчас handoff не должен выглядеть как:

* “я передал задачу”
* “ожидаю ответа”
* “сообщу позже”

Вместо этого каждый handoff должен содержать:

* цель handoff;
* что уже проверено;
* какие evidence собраны;
* что именно требуется от следующего агента;
* критерий завершения подзадачи.

Формат handoff должен быть структурирован и использоваться системой программно, а не просто как текст в чат.

---

# 13. Incident-to-code bridge

Нужно улучшить связку между operational incident и кодовым исправлением.

Если DevOps выявляет вероятную кодовую причину, система должна уметь:

1. сформировать problem package для Developer:

   * suspected service;
   * environment;
   * deployment window;
   * relevant logs;
   * observed failure;
   * suspect components;
   * evidence refs;

2. передать это как task package Developer-агенту;

3. Developer должен:

   * локализовать кодовую проблему;
   * подготовить минимальный фикс;
   * оформить изменения;
   * вернуть structured result.

Эта связка должна быть формальной и воспроизводимой.

---

# 14. Улучшенная approval model

Фаза 1 уже должна была внедрить approve для deploy и опасных действий.
Фаза 2 должна сделать approval layer более зрелым.

Нужно реализовать:

* `approval_request_id`
* `action_type`
* `target`
* `risk_level`
* `rollback_plan`
* `verification_plan`
* `requested_at`
* `approved_at`
* `approved_by`
* `status`

Перед approve система обязана показать пользователю:

* что именно собирается сделать;
* почему это действие предлагается;
* на основе каких evidence;
* какие риски;
* как откатиться;
* как будет проверен результат.

Без этого approve не должен запрашиваться.

---

# 15. Long-running task support

Нужно учесть, что сложные расследования и remediation flows могут быть долгими.

Система должна поддерживать:

* длительные runs;
* возобновление после паузы;
* восстановление состояния после ошибки;
* сохранение текущего прогресса;
* повторный вход в run;
* корректное продолжение после approval от пользователя.

Нельзя, чтобы вся логика зависела от одного линейного chat turn.

---

# 16. Улучшение UI/UX для сложных задач

Чат должен стать интерфейсом наблюдения за execution engine.

Нужно добавить или улучшить:

* отображение root task;
* отображение подзадач;
* статусы задач;
* текущий активный шаг;
* прогресс по плану;
* evidence collected;
* checkpoints;
* ожидаемые действия пользователя;
* final resolution summary.

Нужно предусмотреть две формы отображения:

## 16.1. Human-friendly view

Для пользователя:

* понятно;
* коротко;
* без лишнего шума.

## 16.2. Technical trace view

Для отладки:

* все шаги;
* решения;
* tool calls;
* replans;
* artifacts;
* dependencies.

# 17. Надёжность и устойчивость runtime

Нужно усилить runtime-устойчивость системы:

* recoverable tool failures;
* retries with backoff;
* task retry policy;
* partial failure handling;
* graceful degradation;
* cancellation handling;
* idempotent task resumption where possible.

Если один шаг провалился, система должна уметь:

* повторить шаг;
* отметить задачу как failed;
* перепланировать;
* запросить помощь пользователя;
* завершить run контролируемо.

# 18. Конфигурируемость

Нужно вынести в конфиг или policy layer, а не хардкодить в логике:

* максимальное количество шагов;
* лимиты replanning;
* approval-required action classes;
* allowed action classes by role;
* timeout values;
* retry policy;
* default model params per role;
* task types;
* checkpoint types;
* severity rules;
* confidence thresholds;
* escalation thresholds.

---

# 19. Модель принятия решений: confidence и decision quality

Система должна научиться явно выражать уровень уверенности.

Для ключевых решений поддержать:

* `confidence_score`
* `evidence_count`
* `alternative_hypotheses`
* `decision_reason`

Это нужно минимум для:

* выбора root cause;
* выбора remediation path;
* выбора handoff;
* определения, нужен ли approval;
* определения, нужен ли replan;
* решения, можно ли считать задачу завершённой.

---

# 20. Что НЕ является целью фазы 2

Во избежание расползания scope, в эту фазу **не включать** без необходимости:

* полную автономию без человека в контуре;
* автоматический deploy без approve;
* самообучение агентов;
* fine-tuning моделей;
* внедрение локальных LLM;
* мультимодальность, если это не требуется текущим workflow;
* полную замену существующего UI.

Главная цель фазы 2 — **execution maturity**, а не “AI magic”.

# 22. Обязательные тестовые сценарии

## Сценарий A — Большой incident run

Пользователь:
“Разберитесь, почему сайт не работает, и исправьте проблему”

Ожидается:

* root task;
* decomposition;
* triage subtasks;
* evidence collection subtasks;
* route to infra/code;
* possible code fix subtask;
* approval before deploy;
* verification;
* final closure.

---

## Сценарий B — Replanning

Начальная гипотеза: infra issue
После логов выясняется: code issue

Ожидается:

* зафиксированный replan;
* закрытие/понижение старых гипотез;
* создание новых developer subtasks;
* сохранение evidence trail.

---

## Сценарий C — Approval checkpoint

Система подготовила PR и хочет деплоить фикс.

Ожидается:

* сформирован approval package;
* без approve deploy не происходит;
* после approve execution resumes.

---

## Сценарий D — Task budget exhaustion

Система не может получить прогресс в течение заданного числа шагов.

Ожидается:

* controlled stop;
* краткий отчёт;
* явное указание, что мешает продолжить.

---

## Сценарий E — Resume after interruption

Run был прерван на checkpoint или ошибке.

Ожидается:

* восстановление состояния;
* продолжение с нужного шага;
* без потери task tree и evidence.

---

## Сценарий F — Mixed infra + code issue

Ожидается:

* параллельное или последовательное выполнение подзадач;
* координация между DevOps и Developer;
* единый final summary.

# 23. Definition of Done

Фаза 2 считается выполненной только если:

1. в системе появился полноценный task execution engine;
2. root tasks умеют декомпозироваться в подзадачи;
3. есть task tree и зависимости;
4. работает planner/executor split;
5. система умеет перепланировать действия;
6. есть execution journal;
7. есть artifact store;
8. есть checkpoints и approval-aware resumption;
9. долгие задачи можно продолжать без потери состояния;
10. UI отражает структуру выполнения;
11. система проходит обязательные e2e сценарии;
12. задачи типа “разобраться и исправить” выполняются как многошаговый процесс, а не как набор чатовых реплик.

# 25. Правила выполнения задачи

* Не переписывай систему с нуля без необходимости.
* Максимально используй результаты фазы 1.
* Каждый архитектурный шаг обосновывай проблемой, которую он решает.
* Предпочитай расширение существующей модели выполнения, а не дублирование логики.
* Не оставляй критические решения только на уровне промпта, если их можно закрепить в runtime/state machine.
* Если текущая архитектура мешает фазе 2, сначала зафиксируй это явно и предложи controlled refactor plan.
* После каждой крупной подфазы делай промежуточный отчёт:

  * что сделано,
  * какие файлы изменены,
  * какие риски,
  * что дальше.
